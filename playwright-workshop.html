El Problema
Testing frontend
es doloroso por defecto
🌐

Multi-browser es una pesadilla

Chromium, Firefox y Safari se comportan diferente. Reproducir bugs cross-browser manualmente escala fatal.

⏳

Timing y race conditions

Los tests fallan intermitentemente por sleep() hardcodeados o elementos que aún no existen en el DOM.

🔧

Setup complejo

Selenium requiere WebDriver, ChromeDriver, bindings... demasiada infraestructura antes de escribir el primer test.

🐌

Tests lentos

Cypress corre en el mismo contexto del browser — no puede manejar múltiples tabs, iframes ni contextos de autenticación independientes.

👁️

Debugging opaco

Cuando un test falla en CI, sin capturas, sin red log, sin DOM snapshot — es prácticamente imposible saber qué pasó.

✅

Playwright resuelve todo esto

Un framework que nació aprendiendo de los errores de sus predecesores, construido por el equipo de Chromium en Microsoft.

¿Qué es?
Un framework de testing
end-to-end que controla browsers reales

🦊 Multi-browser nativo

Chromium, Firefox y WebKit (Safari) con la misma API. Un test, tres browsers. Incluye sus binarios — nada que instalar aparte.

⚡ Auto-wait inteligente

Playwright espera automáticamente que cada elemento sea visible, habilitado y estable antes de interactuar. Cero sleep() manuales.

🔀 Paralelo por defecto

Corre tests en paralelo sobre múltiples workers. Los contextos de browser son aislados, rápidos y no comparten estado.

📡 Network interception

Intercepta, modifica o mockea llamadas HTTP/HTTPS en tiempo real. Perfecto para simular errores de API.

🎭 Múltiples contextos

Simula varios usuarios/sesiones en un mismo test. Ideal para flujos colaborativos o multi-rol.

🗺️ Trace Viewer

Graba cada test como una película: screenshots, DOM snapshots, network log, console errors. Debugging visual completo.

Setup
Instalación en 3 minutos
  • 01
    Instalar Playwright Ejecuta el init wizard — detecta tu proyecto y configura todo automáticamente.
  • 02
    Seleccionar browsers El asistente pregunta qué browsers instalar. Los binarios se descargan solos en node_modules.
  • 03
    Configurar baseURL En playwright.config.ts define tu URL base, timeouts y proyectos (browsers).
  • 04
    Correr tests de ejemplo Playwright genera tests de demo. Córrelos para verificar que todo funciona antes de escribir los tuyos.
terminal
# Con npm
npm init playwright@latest

# Con yarn
yarn create playwright

# En proyecto existente (Next.js, Vite...)
npx playwright install

# Instalar solo los browsers
npx playwright install --with-deps
correr tests
# Modo headless (CI)
npx playwright test

# Con UI interactiva
npx playwright test --ui

# Solo un archivo
npx playwright test login.spec.ts

⚙️ playwright.config.ts

El corazón de la configuración. Define browsers (projects), baseURL, timeouts, retries en CI, directorio de tests y opciones del reporter. Siempre empieza aquí.

🧪 *.spec.ts

Convención de nombre para los archivos de test. Playwright los detecta automáticamente. Puedes cambiar el patrón en la config.

📊 playwright-report/

Reporte HTML generado automáticamente tras cada run. Abre con npx playwright show-report. Agréga en .gitignore.

🚀 .github/workflows/playwright.yml

Pipeline de CI pre-configurado para GitHub Actions. Corre tests en todos los browsers en cada push. Listo para usar desde el día 1.

Código
Anatomía de un test real
tests/checkout.spec.ts
import { test, expect } from '@playwright/test';

// Grupo de tests relacionados
test.describe('Checkout flow', () => {

  // Hook: corre antes de cada test
  test.beforeEach(async ({ page }) => {
    await page.goto('/carrito');
  });

  test('agrega producto al carrito', async ({ page }) => {
    await page
      .getByTestId('add-to-cart-btn')
      .click();

    // Assertion con auto-wait
    await expect(
      page.getByRole('status')
    ).toHaveText('1 producto');
  });

  test('mockea el pago con error', async ({ page }) => {
    // Interceptar red
    await page.route('**/api/payment', route =>
      route.fulfill({ status: 500 })
    );
    await page.getByText('Pagar').click();
    await expect(
      page.getByRole('alert')
    ).toBeVisible();
  });
});

🎯 Locators recomendados

getByRole → accesible y resiliente
getByLabel → inputs por su label
getByTestId → data-testid attribute
getByText → por contenido visible

✅ Assertions clave

toBeVisible() · toHaveText()
toHaveURL() · toBeEnabled()
toHaveValue() · toHaveCount()

Encapsula selectores y acciones de cada página en una clase. Hace los tests mantenibles y reutilizables cuando la UI cambia.

Pro Tips
Features que te van a encantar
🔐

Auth state reutilizable

Guarda el estado de autenticación con storageState y reutilízalo en todos los tests. No más login repetido — los tests corren al menos 3× más rápido.

🎬

Trace Viewer

Activa trace: 'on-first-retry' en la config. Cuando un test falla, abre el trace para ver una reproducción completa paso a paso.

📸

Visual regression

Con toHaveScreenshot() Playwright compara capturas pixel por pixel. Detecta cambios visuales no intencionales automáticamente.

🌐

API testing integrado

Usa request para hacer llamadas REST en tus tests. Ideal para crear datos de prueba o limpiar el estado antes de cada test.

📱

Emulación móvil

Emula dispositivos reales con devices['iPhone 14'] — viewport, user agent, touch, geolocation. Sin necesitar dispositivo físico.

🧩

Fixtures personalizados

Extiende test con tus propios fixtures — usuario autenticado, datos mock, helpers. Elimina boilerplate en todos los tests.

CI/CD
Pipeline listo
para GitHub Actions
.github/workflows/playwright.yml
name: Playwright Tests
on:
  push:
    branches: [main, develop]
  pull_request:
    branches: [main]

jobs:
  test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: 20

      - name: Install dependencies
        run: npm ci

      - name: Install Playwright Browsers
        run: npx playwright install --with-deps

      - name: Run Playwright tests
        run: npx playwright test

      - uses: actions/upload-artifact@v4
        if: always()
        with:
          name: playwright-report
          path: playwright-report/
          retention-days: 30

🚀 Generado automáticamente

Al ejecutar npm init playwright el wizard pregunta si quieres el workflow de CI. Si dices sí, lo crea listo para usar.

📦 Cache de browsers

Agrega caché para los binarios de Playwright y reduce el tiempo de CI de ~2 min a ~20 seg en ejecuciones subsecuentes.

📊 Reporte como artefacto

El reporte HTML queda disponible en la sección Artifacts de cada run. Tu equipo puede revisarlo sin acceso al servidor.

🔁 Retries automáticos en CI

Configura retries: 2 en la config para CI. Tests flaky se reintentan automáticamente antes de fallar el pipeline.
